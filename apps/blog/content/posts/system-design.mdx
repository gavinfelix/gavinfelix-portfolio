# 系统设计入门：从问题到架构

**系统设计**（System Design）讨论的是：在给定业务与约束下，如何把多个服务、存储和网络组件组合成**可靠、可演进**的整体。它和「写某个接口」不同，更关注**容量、故障、一致性成本**和**长期维护**。

## 为什么要学系统设计

- **面试**：很多公司会用开放题考察你是否能拆解需求、做合理取舍，而不是背标准答案。
- **工作**：需求从「单机够用」变成「多机房、多团队」时，早期架构决定后期改动的成本。
- **沟通**：用同一套词汇（可用性、延迟、一致性模型）和上下游对齐预期。

## 一个常用的思考框架

没有唯一模板，下面这套在面试和文档里都很常见：

1. **澄清需求**：功能范围（读多写多？实时还是最终一致？）、非功能指标（QPS、延迟、可用性目标）。
2. **粗略估算**：流量、存储、带宽的数量级，避免数量级错误。
3. **API 与数据模型**：对外接口长什么样、核心实体与关系。
4. **高层架构**：客户端、网关、应用服务、缓存、数据库、消息队列如何协作。
5. **深入关键点**：热点、单点、一致性与事务边界。
6. **权衡与演进**：当前方案的代价、未来如何拆分或扩容。

每一步都可以反问：**如果流量翻 10 倍，哪里会先撑不住？**

## 核心概念（速览）

### 可扩展性（Scalability）

- **垂直扩展**：更强单机；简单但有上限、成本高。
- **水平扩展**：更多节点；需要**无状态服务**、**数据分片**与**协调机制**（如负载均衡、分布式 ID）。

### 可用性（Availability）与容错

常用指标是 **SLA**（例如 99.9% 与 99.99% 的允许停机时间差一个数量级）。手段包括：**冗余**、**健康检查与摘除故障节点**、**限流与降级**、**异步削峰**。

### 一致性与 CAP

- **强一致**：读写立刻反映全局状态；往往牺牲延迟或可用性。
- **最终一致**：允许短暂不一致；适合通知、统计、搜索索引等场景。

CAP 不是说只能三选二，而是：**分区发生时**，你更倾向保证**一致性**还是**可用性**，要按业务选择。

### 延迟与吞吐

- **延迟**：单次请求端到端时间；受网络、序列化、锁、磁盘与下游依赖影响。
- **吞吐**：单位时间处理的请求量；可通过并行、批处理、缓存命中提升。

优化前先**测量**：找到真正的瓶颈，而不是凭感觉加机器。

## 常见构建块

| 组件 | 典型用途 |
|------|----------|
| **负载均衡** | 把流量分到多实例，配合健康检查 |
| **缓存** | 减轻读库压力；注意失效策略与缓存击穿 |
| **CDN** | 静态资源就近分发，降低延迟与源站压力 |
| **关系型 DB** | 事务与复杂查询；注意主从、读写分离与分库分表 |
| **NoSQL / 宽表** | 高吞吐写入、灵活 schema；牺牲部分查询能力 |
| **消息队列** | 解耦、异步、削峰；注意重复消费与顺序性 |

这些不是「越多越好」。每多一层，**运维复杂度、故障排查路径**都会变长。

## 小结

系统设计本质是**在约束下做权衡**：没有「满分架构」，只有**是否匹配当前阶段的目标**。练习时可以多画**数据流与故障场景**（某台机器挂了、某库慢了会怎样），比死记名词更有用。
